  <action>
Add `customLovelaceModules` to the existing `services.home-assistant` block in `modules/automation/home-assistant.nix`:

```nix
customLovelaceModules = with pkgs.home-assistant-custom-lovelace-modules; [
  advanced-camera-card  # v7.6.5 -- renamed from frigate-hass-card in v7.0.0
];
```

Place this after the existing `customComponents` declaration.

Next, add a Lovelace YAML dashboard configuration. HA supports multiple dashboards: the default one stays in storage mode (UI-editable), and we add a new "cameras" dashboard in YAML mode. This is done via `services.home-assistant.config.lovelace`:

```nix
lovelace = {
  mode = "storage";  # Keep default dashboard UI-editable
  dashboards = {
    cameras = {
      mode = "yaml";
      filename = "cameras-dashboard.yaml";
      title = "Cameras";
      icon = "mdi:cctv";
      show_in_sidebar = true;
      require_admin = false;
    };
  };
  resources = [
    {
      url = "/local/community/advanced-camera-card/advanced-camera-card.js";
      type = "module";
    }
  ];
};
```

Note on resources: When using `customLovelaceModules`, HA NixOS module auto-registers the JS resources. However, declare it explicitly in `lovelace.resources` as a safety measure. If HA auto-registers it AND we declare it, there may be a duplicate -- test after deploy and remove the explicit `resources` block if the card loads without it.

IMPORTANT: The `lovelace.resources` registration approach depends on the NixOS HA module version. The NixOS module for HA automatically adds resources from `customLovelaceModules` to `/local/nixos-lovelace-modules/`. So the resource URL may need to be `/local/nixos-lovelace-modules/advanced-camera-card.js` instead. Check after deploy and adjust.

Next, create the dashboard YAML file via tmpfiles. Add to the existing `systemd.tmpfiles.rules`:

```nix
"f /var/lib/hass/cameras-dashboard.yaml 0644 hass hass"
```

This creates an empty file so HA does not crash on boot. The actual dashboard YAML content will be written in Task 2.

Also add a `lovelace_resources.yaml` tmpfile if needed for resource declaration persistence:

```nix
"f /var/lib/hass/lovelace_resources.yaml 0644 hass hass"
```

Run `make check` to verify the configuration builds without errors. If `lovelace.resources` causes a build error (schema mismatch), remove it -- the NixOS module handles resource registration automatically for `customLovelaceModules`.

<task type="auto">
  <name>Task 2: Write cameras dashboard YAML and deploy to ser8</name>
  <files>modules/automation/home-assistant.nix</files>
  <action>
Write the cameras dashboard YAML content. Since HA YAML-mode dashboards read from a file, we need to deploy the dashboard content to `/var/lib/hass/cameras-dashboard.yaml` on ser8.

The approach: use `systemd.tmpfiles.rules` with a `C+` or `L+` copy rule, or use `environment.etc` to place a managed file. The most NixOS-idiomatic approach is to write the dashboard YAML inline in the tmpfiles rule using the `f+` type with content:

```nix
# In systemd.tmpfiles.rules, replace the empty file rule with content:
"f+ /var/lib/hass/cameras-dashboard.yaml 0644 hass hass - ${dashboardYaml}"
```

However, tmpfiles content is limited. The better approach is to write the YAML to a Nix store path and copy it:

Create a `let` binding at the top of the module:

```nix
let
  camerasDashboardYaml = pkgs.writeText "cameras-dashboard.yaml" (builtins.toJSON dashboardConfig);

  dashboardConfig = {
    views = [
      # Tab 1: Live Cameras
      {
        title = "Live Cameras";
        path = "live";
        icon = "mdi:camera";
        type = "panel";
        cards = [
          {
            type = "vertical-stack";
            cards = [
              # 3-camera grid using advanced-camera-card
              {
                type = "grid";
                columns = 3;
                square = false;
                cards = [
                  # Driveway camera
                  (cameraCard "driveway" "Driveway")
                  # Front Door camera
                  (cameraCard "front_door" "Front Door")
                  # Garage camera
                  (cameraCard "garage" "Garage")
                ];
              }
              # Detection Controls section below camera grid
              {
                type = "horizontal-stack";
                cards = [
                  {
                    type = "entities";
                    title = "Detection Controls";
                    show_header_toggle = true;
                    entities = [
                      { entity = "switch.driveway_detect"; name = "Driveway Detection"; }
                      { entity = "switch.front_door_detect"; name = "Front Door Detection"; }
                      { entity = "switch.garage_detect"; name = "Garage Detection"; }
                      { entity = "switch.driveway_motion"; name = "Driveway Motion"; }
                      { entity = "switch.front_door_motion"; name = "Front Door Motion"; }
                      { entity = "switch.garage_motion"; name = "Garage Motion"; }
                    ];
                  }
                ];
              }
            ];
          }
        ];
      }
      # Tab 2: Events
      {
        title = "Events";
        path = "events";
        icon = "mdi:motion-sensor";
        cards = [
          # Driveway events
          (eventCard "driveway" "Driveway Events")
          # Front Door events
          (eventCard "front_door" "Front Door Events")
          # Garage events
          (eventCard "garage" "Garage Events")
        ];
      }
    ];
  };

  # Helper: advanced-camera-card for a single camera
  # Per user decision: live video stream default, click to fullscreen, no bounding boxes on live
  cameraCard = cameraName: displayName: {
    type = "custom:advanced-camera-card";
    cameras = [
      {
        camera_entity = "camera.${cameraName}";
        live_provider = "auto";
        title = displayName;
        frigate = {
          camera_name = cameraName;
        };
      }
    ];
    live = {
      controls = {
        builtin = true;
      };
    };
    menu = {
      buttons = {
        frigate = { enabled = true; };
        fullscreen = { enabled = true; };
      };
    };
    view = {
      default = "live";
    };
  };

  # Helper: advanced-camera-card in event/gallery mode for event browsing
  # Per user decision: thumbnail grid, filterable, 7-day history, inline expand
  eventCard = cameraName: displayName: {
    type = "custom:advanced-camera-card";
    cameras = [
      {
        camera_entity = "camera.${cameraName}";
        title = displayName;
        frigate = {
          camera_name = cameraName;
        };
      }
    ];
    view = {
      default = "clips";
    };
    event_gallery = {
      min_columns = 3;
    };
  };
in
```

IMPORTANT: The `builtins.toJSON` approach converts Nix attrsets to JSON. HA Lovelace YAML mode actually reads YAML, but JSON is valid YAML. However, `type = "panel"` view with a vertical-stack may have layout issues. If `panel` mode does not support grid properly, use `type = "masonry"` or remove `type = "panel"` for a standard multi-card view.

Alternative approach if `builtins.toJSON` causes issues with HA YAML parsing: use `pkgs.writeText` with a raw YAML string using Nix multiline strings (`''...''`).

After creating the dashboard config in Nix, add a systemd oneshot service or tmpfiles rule to copy the dashboard file to `/var/lib/hass/`:

```nix
systemd.tmpfiles.rules = [
  # ... existing rules ...
  "L+ /var/lib/hass/cameras-dashboard.yaml 0644 hass hass - ${camerasDashboardYaml}"
];
```

Actually, `L+` creates a symlink. Since HA may need to write to this file in some modes, use `C+` (copy with replace) instead:

```nix
"C+ /var/lib/hass/cameras-dashboard.yaml 0644 hass hass - ${camerasDashboardYaml}"
```

Or even simpler, use the existing pattern from this codebase -- a systemd service that copies config files (like `media-config.service` on ser8). But that is overkill for one file. The tmpfiles `C+` approach is simplest.
